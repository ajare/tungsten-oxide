# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A browser-based, dependency-free (aside from three.js via CDN) racing track editor and driving game, built as plain HTML/JS with no build step. There is no root `package.json`, no bundler, no test runner for the top-level app â€” files are opened/served as static assets.

- `track.html` â€” the driving game. Loads three.js (CDN), `track-core.js`, `js/track-game.js`.
- `editor.html` â€” the 2D/elevation track editor UI. Loads `track-core.js`, `js/editor.js`.
- `index.html` â€” unrelated scratch demo (spinning cube in three.js), not part of the track app.
- `track-core.js` â€” shared track math, used by both the game and the editor (see architecture below).
- `js/track-game.js` â€” three.js scene, track mesh generation/rendering, car physics/controls, collisions.
- `js/editor.js` â€” editor state, undo/redo, canvas rendering/interaction for authoring tracks.
- `js/track-mesh.js` â€” shared mesh-region math (see below). The mesh-world counterpart to `track-core.js`, split out because it depends on geometry-js while `track-core.js` stays dependency-free.
- `ext/geoemetry-js/` â€” a git submodule (`@willpower/geometry`, https://github.com/ajare/geoemetry-js), a separate self-contained ES-module mesh/geometry library with its own `package.json`, tests, and a React/Vite editor. It's linked into the root project as a local npm dependency (root `package.json` -> `"@willpower/geometry": "file:ext/geoemetry-js"`, installed via `npm install`, resolved as a symlink at `node_modules/@willpower/geometry`) so track code can `import` it as `@willpower/geometry`. See `ext/geoemetry-js/README.md` for its own commands (`npm test`, `npm --prefix editor/ui run dev`, etc.) and its own codebase map.

## Running / testing

No build step: open `track.html` or `editor.html` directly, or serve the repo root statically. Run `npm install` once (after `git submodule update --init --recursive`) to link the `@willpower/geometry` local dependency.

- `npm test` â€” Node's built-in runner over `test/*.test.js` (pure logic: mesh geometry, rail collision, schema round trips). Fast, no browser.
- `node tools/browser-smoke.mjs` â€” drives the real pages in headless Chromium. Catches ESM/import-map breakage, runtime errors and physics regressions the unit tests can't see. Needs `npm install --no-save playwright && npx playwright install chromium`. Deliberately outside `test/` so `node --test` doesn't try to run it.

For the `geoemetry-js` submodule, its own commands apply (`npm test`, `npm --prefix ext/geoemetry-js/editor/ui run dev`) â€” see `ext/geoemetry-js/README.md`.

**The `.js` files are ES modules** (root `package.json` has `"type": "module"`), except `track-core.js`, which is deliberately a classic browser script â€” an IIFE assigning `window.TrackCore`. Tests load it by evaluating its source with a stand-in `window`, not by importing it.

## Architecture: track data model (`track-core.js`)

This is the core concept to understand before touching the game or editor â€” read the file header comment in `track-core.js` for the authoritative spec. Summary:

A **track** is `{ version, name, samples, paths: [...], disjointSeams: [...], start: {...} }`. Each **path** is either a closed loop or an open curve, and holds one ordered array of **typed control points**:

```
points: [
  { type: 'position', id: 'p1', pos: [x,y,z], weight },
  { type: 'roll',     t: 0..1, roll: <deg> },
  { type: 'width',    t: 0..1, width: <full width> },
  { type: 'crossSection', t: 0..1, curvature, tightness, thickness },
  ...
]
```

A `crossSection` point shapes the road's profile *across* its width: `curvature` (âˆ’1..1) crowns or dishes it, `tightness` is the exponent on that arc, and `thickness` extrudes the whole profile **downward into a shell** with an underside and side walls (0 = the original zero-thickness sheet). The shell is purely visual â€” physics still projects onto the top surface only. `curvature` and `tightness` are scale-invariant; `thickness` is a length and scales with the world's units.

The game and the USD exporter each build the shell from `TrackCore.crossSectionHeight` plus the frame's normal and thickness, in their own mesh/prim separate from the road surface â€” which keeps the road's UV mapping untouched and gives the substructure its own material. In the editor, every cross-section point is minted through `crossSectionPoint()` in `js/editor.js`: splits, joins, seam reconnects, type conversions and insertions all construct these, and spelling the fields out at each site is how a new field gets silently dropped (a split would quietly reset thickness to the default).

- Each control-point type is independent â€” its own count, its own spacing â€” and only interacts with points of its own type.
- `position` points interpolate with a rational, uniformly-knotted cubic B-spline (NURBS); their order in the array *is* the path's shape sequence.
- `roll`/`width` points each interpolate with their own non-uniform Catmull-Rom/Hermite spline over their own `t` (a fraction of the path's parameter domain, independent of array order).
- All wrap for closed paths, clamp at the ends for open ones. Positive roll lifts the LEFT edge (banks into a right-hand turn).
- Use `TrackCore.splitPoints(path.points)` to get the three filtered, t-sorted arrays (`controlPoints`, `rollPoints`, `widthPoints`) that the math functions actually consume â€” these are filtered *views* (same objects, not copies), so holding onto one and later splicing it out of `points` is safe.
- Position point `id`s are stable editor identities. If the same position ID appears in multiple path occurrences, `parseTrack()` unifies them into the same in-memory object, so editing one moves every occurrence. `disjointSeams` lets the editor reverse hard-corner split/open operations; the game only needs point IDs plus seam pointIds to cut disjoint edges.
- `start: { path, point, reverse }` picks which position control point the player begins at (nearest baked sample) and facing direction relative to the path's natural parametric direction.

Public API exposed as `window.TrackCore`: `basis`/`basisDeriv` (B-spline basis + derivative), `splitPoints`, `makeEvaluator(cps, closed)` â†’ `{ evalTrack(g), CP_N, closed }`, `buildCenterline`, `buildEdges`, `parseTrack`/`serializeTrack`, `cloneTrack`, `crossSectionHeight`/`crossSectionHeightDerivative`, `crossSectionBreakpoints`, `DEFAULT_TRACK`/`STARTER_TRACK`, `N_DEFAULT`.

Both `js/track-game.js` (3D mesh/physics) and `js/editor.js` (2D authoring UI) build on top of this same shared math so the editor's preview and the game's actual track geometry can never drift apart â€” if you change interpolation/eval behavior, change it once in `track-core.js`.

**`js/usd-export.js` is a third consumer of that same math**, and it is the one that can drift unnoticed, because nothing on screen shows an exported file being wrong. It had its own copy of the cross-section profile â€” a triangular tent where the game and editor build a semicircular arc â€” so any track with a curved cross section exported to a shape nobody authored. The profile now lives in `TrackCore.crossSectionHeight`, and a test compares exported vertices against it directly. Note *why* that test uses intermediate `v` values: every plausible profile formula agrees at the road's centre and both edges, so sampling only those three points cannot catch this class of bug.

### Adaptive cross-section tessellation

The mesh across a ring's width is **not** a fixed segment count: `TrackCore.crossSectionBreakpoints(curvature, tightness, chordWidth)` picks each ring's own `v`-breakpoints by recursively bisecting any cell whose true profile deviates (chord-sagitta, at the cell's midpoint) from a straight line between its two ends by more than a fixed world-unit tolerance, capped at a fixed recursion depth so a profile with near-infinite slope at the edges (see `crossSectionHeightDerivative`'s comment) can't blow up the triangle budget. A flat ring (`curvature = 0`) collapses to just its two edges â€” one quad per longitudinal strip, no wasted geometry on straight road.

Because `curvature`/`tightness` are authored per cross-section point and interpolate along a path, **adjacent rings can pick different breakpoints** â€” the classic T-junction/gap risk. Both `js/track-game.js`'s `buildPath` and `js/usd-export.js`'s `buildCurveMesh`/`buildCurveShellMesh` fix this the same way: for each longitudinal strip, take the **union** of the two rings' breakpoint sets, so neither ring's edge in that strip is a wide, unsplit segment against the other's finer one.

**The union alone is not sufficient, and shipped as a real bug before this was caught.** A ring is shared by *two* strips â€” its left neighbor and its right neighbor â€” and each strip computes its *own* union. If both strips then evaluate the ring's true analytic surface (`crossSectionHeight`) at whatever "foreign" `v`s their own union happens to include, the SAME ring ends up drawn as two different polylines: one strip might place a real vertex at, say, `v=0.25` (because that ring's other neighbor needed it there), while the other strip's union never introduces that point and instead spans it with a single straight chord. Both are individually "exact" samples of the true curve, but a chord and a polyline through an extra point on the same curve are not the same 3D shape â€” the gap between them is exactly the visible seam. Evaluating the true surface at *any* v that the ring didn't choose for itself is the bug, not a rounding error: it can be several world units wide on an ordinarily-sized road, not a sliver.

The fix is `TrackCore.crossSectionStitchPoint(ownBreaks, v, pointAt)`: a `v` already in the ring's own breakpoints returns the true point; any other `v` is **linearly interpolated between the ring's own two real bracketing vertices** instead of resampled from the curve. A point lerped between two vertices that are already part of a ring's polyline is collinear with the straight segment already there, so inserting it changes nothing about how that ring looks â€” meaning the ring's rendered edge is bit-for-bit identical no matter which neighbor, or how many foreign `v`s, ask for it. Both `js/track-game.js` (`ringPoint`/`ringUnderPoint`) and `js/usd-export.js` (same names) route every union-driven vertex through this, never through the raw per-ring surface/under point function directly. The same per-ring breakpoints also drive the shell's underside, side walls and end caps, so the shell always tessellates identically to the top surface above it. `crossSectionBreakpoints`/`crossSectionStitchPoint` live in `track-core.js` (pure, unit-tested); the union-stitching and mesh-assembly code is consumer-specific and duplicated between the game and the exporter on purpose (three.js `BufferGeometry` vs. USD's `points`/`faceVertexIndices`).

### Adaptive longitudinal ring spacing (mesh-only)

The same idea applies **along** the path, not just across it: `TrackCore.buildAdaptiveMeshFrames(controlPoints, closed, rollPoints, widthPoints, crossSectionPoints, raw, edges)` takes the already-baked physics `raw`/`edges` (uniform-in-parameter frames from `buildCenterline`, at a per-path sample count `TrackCore.adaptiveSampleCount` picks from the path's driven length â€” see below) and produces a **separate, mesh-only** frame/edge array with denser rings on sharp turns/hills and sparser ones on long straights â€” pure rendering, consumed only by `js/track-game.js`'s `buildPath` for the road surface + shell. **Physics/collision never sees this mesh-only resampling.** `centerline`, `computePhysicalWallOffsets`, `sLeft`/`sRight`, and guard rails all ride on the physics baking, untouched by the mesh pass â€” deliberately, because physics here is genuinely frame-based (unlike cross-section height, which physics gets from a continuous analytic function).

**Two different adaptivities, do not conflate them.** The *physics* baking is **uniform-in-parameter** with a per-path COUNT that scales with track length: `TrackCore.adaptiveSampleCount(controlPoints, closed, rollPoints, widthPoints, crossSectionPoints)` measures the driven length and returns `clamp(round(length / 6 m), N_DEFAULT, 2000)`, so frame *spacing* stays ~constant (~6 m) as tracks grow from ~1 km to 7â€“10 km â€” which is exactly what keeps the tuned tolerances (`SEGMENT_ALONG_TOL`, `CORRIDOR_ALONG_TOL`) valid at any track size. `js/track-game.js`'s `buildTrack` calls it per path and feeds the SAME count to `makeSelfIntersectionDeciders` (its frameâ†’control-id mapping is relative to that count). This is distinct from N_DEFAULT/`track.samples`, which stay the USD-export ring count and the floor here â€” **the USD exporter is not adaptive and is untouched by this.** The editor deliberately keeps its 2D preview + self-intersection scan at the fixed `N_DEFAULT` (its idle O(NÂ²) crossing detection would hitch at Nâ‰ˆ2000, and a 400-sample preview of the same curve is visually identical); the crossing-override keys are control-point-id pairs, resolution-independent, so editor-at-400 and game-at-adaptive-N still resolve the same overrides. The *mesh* pass here, by contrast, is **adaptive-in-curvature**, packing rings onto sharp turns/hills; it never feeds back into physics.

The trigger is chord-sagitta of the true 3D position (`TrackCore.longitudinalBreakpoints(g0, g1, posAt)`, mirroring `crossSectionBreakpoints` but along `g` instead of across `v`) â€” deviation of a cell's true midpoint from the straight-line midpoint between its two ends. This naturally captures turns, hills and banking-induced bending all at once, with no separate curvature formula needed. Unlike the cross-section version there is also a **max-distance cap** (`LONGITUDINAL_MAX_DISTANCE`, 40 world units): a perfectly straight run has zero sagitta forever, so without a hard cap an arbitrarily long straight would collapse to just its two endpoints, stretching UV mapping and letting roll/width/thickness drift unbounded between two samples even where position itself is dead straight. A cell splits when its sagitta exceeds `LONGITUDINAL_SAGITTA_TOLERANCE` (`0.1`, same value and reasoning as the cross-section tolerance) **or** its chord length exceeds the distance cap, capped at `LONGITUDINAL_MAX_DEPTH` recursion levels as a defensive backstop.

**Self-intersection trimming (`trimEdge`/`removeLocalEdgeSelfIntersections`) is mandatory, not optional, and this only works because it is never re-run.** Trimming collapses a tight inner corner into a sharp mitred point; skip it and any sufficiently tight turn renders as an ugly self-overlapping mesh. If the mesh resampled its own frame positions independently of physics, it would need its own trim pass â€” and a mesh-only trim, computed from a different sample set than physics's, could disagree with the physics corridor exactly at the corners where it matters most. Instead, `buildAdaptiveMeshFrames` detects which of physics's original frames actually had their edge repositioned by trimming (`foldAffectedIndices`, comparing the real edge against the plain untrimmed half-width offset â€” trimming moves a whole folded run to one shared point, so any real change is well outside float noise) and carries those frames through **byte-for-byte, verbatim**. Adaptive resampling only ever touches the interior of a fold-free run between two such frames, where it's free to both add rings (sharp bends) and remove them (straights) relative to the original baking. This also transparently preserves disjoint-seam endpoint overrides (`endpointNormals`/`endpointCuts` in `buildPath`), since those move a frame's edge away from the untrimmed formula exactly the same way trimming does.

There is no cross-section-style stitching/crack risk here: longitudinal frames form one sequential chain, not a grid shared between two independent perpendicular strips, so the same frame is never drawn two different ways by two different "neighbors."

## Architecture: mesh regions (`js/track-mesh.js`)

Flat, drivable areas imported from the geometry-js editor â€” plazas, junction pads, arenas â€” for shapes a swept spline ribbon cannot express. Introduced in schema 4:

```
meshAssets: { <assetId>: { name, railHeight, mesh } }   // mesh = pristine geometry-js JSON
meshes:     [ { id, asset, x, z, rotation, elevation } ] // rigid placements
```

Design rules, all of which the code depends on:

- **Assets are geometry, placements are transforms.** A placement is rigid: X/Z translation, yaw rotation, and one `elevation`. There is no scale, and a region is always horizontal (normal `+Y`, no banking or cross-section). Because the transform is rigid, **a triangulation computed in asset space stays valid for every placement** â€” assets are triangulated once, never per frame.
- **Rail flags live on the asset**, as `attributes.rail` on geometry-js edges, so every placement of a shape is railed identically. Need different railing? Import the file again â€” imports always mint a new asset id (`pad`, `pad-2`), never disturbing existing placements.
- **Two import routes, one code path.** `parseMeshJSON()` validates and returns `{ mesh }` or `{ error }` without ever throwing or touching `track`, so a bad paste is reported before anything is mutated; `addMeshAsset(mesh, name, at)` then mints the asset and placement. File import (`Import Mesh`) centres the region on the view so an asset authored far from `(0,0)` still lands somewhere visible; clipboard import (`Paste Mesh`) places it at the world origin, preserving authored coordinates. `navigator.clipboard` needs a secure context, so the paste path degrades to an alert pointing at the file route. The Edit-mode right-click menu offers a third entry point, centring the region on the click; it opens *immediately* with the paste option hidden and reveals it once an async `clipboardMayHaveText()` resolves. A `menuToken` guards against a slow resolve from an already-dismissed menu revealing the option on a later one.
- **Never call `readText()` just to decide what to show.** An ungranted read pops the browser's own native "Paste" confirmation bubble, so probing the clipboard on right-click means a system popup the user never asked for, layered over our menu, every time â€” and a second one when the paste actually runs. `clipboardMayHaveText()` consults `navigator.permissions.query({name:'clipboard-read'})` first, which is silent: `granted` probes for real, `denied` omits the option, and `prompt` offers it *unprobed* so the one and only read happens on the click that asks to paste. Pinned by a browser test that runs with no clipboard permission â€” granting it in a test hides the entire bug, which is exactly how this shipped twice.
- **Import rails every rim edge** (`TrackMesh.railBoundaryEdges`), so a fresh region is enclosed and drivable immediately; you unrail edges to open ledges. "Rim" means `edge.polygons.size === 1` â€” an edge owned by two polygons is an interior seam you drive across, and railing it would wall a region down the middle. Hole rims are owned by the one polygon holding the hole, so they count as rim and an imported hole starts as a walled pillar rather than a pit.
- **A railed edge is a solid, finite-height wall**; an unflagged boundary edge is a **ledge** you drive off into the existing ballistic code. Holes follow exactly the same rules, so a bare hole is a pit you fall through.
- **Rails are collision everywhere now.** `G` is a pure rendering toggle and never changes what stops the ship.
- **Surface precedence is nearest-in-Y**, not simple containment. `surfaceOwnerAt()` picks whichever of the mesh region and the spline corridor sits closer to the ship's current Y, which is what lets a mesh flyover pass over a ribbon without hijacking the ship underneath.
- **`corridorContains()` is the real containment test, not `projectToSurface()`.** `projectToSurface` returns only a *lateral* offset `s`; a point far off the *end* of a segment projects onto that segment's clamped endpoint and reports a small, in-range `s` while being hundreds of units away. Ignoring this teleports the ship onto a distant ribbon when it leaves a mesh ledge. Always pair the lateral bounds with the along-tangent check.

### Lateral offset is not containment â€” the recurring trap

This has now caused two separate bugs, so it is worth stating as a rule: **anywhere you test whether a point is "on" a piece of track, a lateral/`s` bound alone is wrong.** Because `t` is clamped to `[0,1]`, a point beyond a segment's end projects *onto that end* and inherits its lateral offset â€” which, for a ship running down the middle of a straight road, is `0` for every segment on the path no matter how distant.

Two places guard against it, and both are load-bearing:

- `SEGMENT_ALONG_TOL` in `sampleTrack()`'s `bestUnder` test. Without it, a point past an **open curve's end** is claimed by a far-back segment, so `best` is never the terminal segment, `offEnd` can never fire, and the ship is reprojected backwards instead of launching off the end â€” it becomes impossible to leave a curve. Covered by a browser test that verifies both ends.
- `CORRIDOR_ALONG_TOL` in `corridorContains()`, for the mesh-exit and airborne-landing paths.

Physics on a region is genuinely different code from the corridor: free 2D integration plus swept segment collision (`slideAlongRails`), because there is no lateral parameter to clamp on an arbitrary polygon. Collision is swept, not positional, so a fast ship cannot tunnel through a wall.

**Known limitation, by design:** a region is flat, so it only meets a ribbon cleanly where that ribbon is level and unbanked. Every other join is a visible step. Place regions accordingly.

## World units and schema migration

**Schema 5 doubled the world's unit scale.** Every length in a track â€” control point positions, widths, elevations, rail heights, mesh geometry â€” is twice what the same track measured under schema 4, and the game's ship, speeds, gravity, camera and thresholds were all scaled to match. **Nothing about how a track looks or drives changed; only the absolute units did.** The HUD's `speed Ã— 4.5` factor was halved from `Ã— 9` for exactly this reason, so the km/h readout is unchanged.

Rules if you touch this:

- **Only lengths scale.** Angles (`roll`, `rotation`), curve parameters (`t`), NURBS `weight`, cross-section `curvature` (dimensionless) and `tightness` (an exponent) are scale-invariant. Scaling any of them changes a track's *shape*, not its size. Likewise, rates (`turnRate` rad/s, `grip`, lerp factors, the bob frequency, the landing spring's `-55`) and ratios (`0.98` scrub, `1.5` step factor) stay put â€” only their length/velocity *caps* scale.
- **`scaleRawTrackData()` runs before normalization, deliberately.** Normalization injects defaults (`DEFAULT_WIDTH`, `DEFAULT_RAIL_HEIGHT`) that are already in current units; scaling afterwards would double those too and silently widen every old track that never authored an explicit width. There's a test pinning this.
- **Built-in tracks are authored in current units** and carry `version: TRACK_SCHEMA_VERSION`, so `cloneTrack(DEFAULT_TRACK)` â€” which bypasses `parseTrack` entirely â€” is never re-migrated.
- Migration is keyed off the file's `version` and is idempotent across save/load.
- **The unit migration is keyed to `UNIT_SCALE_SCHEMA_VERSION` (5), not to `TRACK_SCHEMA_VERSION`.** Those were the same number while 5 was current, so `sourceVersion < TRACK_SCHEMA_VERSION` looked correct â€” but it meant the *next* version bump, for whatever unrelated reason, would silently re-double every schema-5 track ever saved. Schema 6 was that bump. If you add a schema version, do not re-point this at the current version; there is a test pinning it.

**Schema 6 added cross-section thickness** â€” an extrusion distance that turns the road ribbon into a shell. It changes no units. Because thickness is a *length*, it scales with the world (`scaleRawTrackData` handles it) while `curvature` and `tightness` beside it do not.

**Schema 7 added zones** (see below). It changes no units, and â€” importantly â€” `scaleRawTrackData` deliberately does **not** touch zones: a zone can only exist in a schema-7+ file, and the unit migration only ever runs on `sourceVersion < 5`, so there is nothing to scale. Every length in a zone (width, length, lateral offset, mesh x/z) is already in current units.

## Architecture: zones (`track.zones`)

Flat rectangular areas that float just above a surface and fire an effect when the ship drives onto them (schema 7). A zone is `{ id, effect, width, length, host, ...effectParams }`:

- **`host` is one of two kinds.** `{ kind:'path', pathId, t, lateral }` anchors it to a spline path â€” centered at `t` (0..1 of the path domain) with a signed `lateral` offset in units, oriented along the track and conforming to its curve/banking. `{ kind:'mesh', meshId, x, z, rotation }` anchors it to a mesh region â€” a flat rotated rectangle at world `x/z`, taking its `y` from the region's elevation. Host ids are the stable path/placement ids; a zone whose host id is gone is dropped (`normalizeZones`, same policy as a dangling mesh placement).
- **`width` is the across-track extent, `length` the along-track extent**, both world units. `effect 'velocityChange'` carries `{ factor, duration }` (boost); `'startGrid'` is a purely visual marker with no mechanical effect for now.
- **`TrackCore.zonePathStrip()` is the shared geometry** for a path zone â€” it walks the evaluator outward from the center `t` accumulating true 3D arc length (interpolating the final partial step so the strip spans exactly `length`, even for a short pad on a big loop), and returns two surface-conforming edge polylines **plus the evaluator g-window `[gLo, gHi]`**. The editor preview and the game mesh both build from it, so they can't drift. The game keeps the g-window and reuses `TrackCore.zoneAlongContains()` (which handles the closed-path wrap) for detection.
- **Boost mechanics live entirely in `js/track-game.js`** and are separate from rendering. A boost fires the frame the ship *enters* a pad while **grounded on that zone's host surface** (a `wasInside` edge per compiled zone; a global `boostActive` lock ignores further triggers until the current one ends). It kicks speed to `factor Ã— maxSpeed` and raises the speed clamp (`effectiveMaxSpeed`) to that for `duration` s, then eases the cap back to normal max over `ZONE_RELEASE` (1 s). During a boost `speedRatio` is clamped to `[0,1]` so the over-max speed doesn't overdrive steering/grip/lean; the HUD shows the true speed plus a BOOST tag. `resetShip`/`respawn` clear the boost.
- **Detection reads the ship's evaluator parameter g** via `shipParamG`, recovered from the segment (`a`/`b`/`segT`) `sampleTrack` records on `_sample` â€” the frameâ†’g mapping matches `buildCenterline`, so it lines up with the zone's own window. Path zones require the ship on that path (`_sample.pathObj === z.hostPath`) and within the along-window + lateral band; mesh zones require the ship on that region and inside the rotated rectangle (`worldToLocal` matching the render corners).
- **Editor UX:** right-click â†’ **Add zone â–¸ Boost / Start grid** anchors to the mesh region under the click or else the nearest path. Zones are hit-tested in Edit mode after control-point/physics handles but before the large mesh regions, selected/dragged (path zone drag re-projects to `t`+`lateral`; mesh zone drag moves `x`/`z`), and edited in the props panel. `ensureTrackIds`/`removeStaleSeams` prune zones whose host is gone.
- **Start-grid zones are checkered.** The game builds a black/white `CanvasTexture` (`makeCheckerTexture`, `NearestFilter`, `RepeatWrapping`) with UVs scaled to `2 Ã— ZONE_CHECKER` world units so checks are square-in-world regardless of zone size (mesh zones use local x/z; path zones use cumulative along-strip distance / width). A fresh texture per start-grid mesh, because `disposeObject` frees `material.map` on rebuild. The editor preview uses a screen-space checker `CanvasPattern`.

## Architecture: triggers (`track.triggers`)

Vertical gate quads the ship passes **through** (schema 8), as opposed to zones (flat, driven over). `{ id, type, host, width, height, rotation, direction }`:

- **Frame is banking-aligned** â€” the gate's normal is the track tangent (`fwd`), width runs along the banked `edgeRight` (`right`), height along the surface normal (`up`), yawed by `rotation` about `up`. `TrackCore.triggerPathFrame` bakes this once (shared by the game's detection/debug render and the editor's top-down line); mesh triggers are a flat gate at the region's elevation whose normal comes from `rotation`. Centered on the centerline (no lateral offset).
- **Host** is `{kind:'path', pathId, t}` or `{kind:'mesh', meshId, x, z}`; `normalizeTriggers` drops dangling hosts. `direction âˆˆ {both, forward, backward}` relative to `fwd`; `type` is `'dummy'` (no effect â€” firing only `console.log`s + flashes the debug view).
- **Detection (`detectTriggers`, `js/track-game.js`, every sub-step, grounded or airborne)** is a **swept** crossing of the ship segment `_prevTriggerPos â†’ groundPos` against the gate plane: on a sign-flip of the signed distance to the plane, the crossing point must be within `|right| â‰¤ width/2` and `0 â‰¤ up â‰¤ height`. Fires once while `armed` (direction permitting) then disarms; **re-arms** when the ship is off the widthÃ—height footprint *or* more than `TRIGGER_REARM_MARGIN` past the plane. `resetTriggers` (on reset/respawn) reseeds `_prevTriggerPos` so teleports never fire, and re-arms all.
- **Not rendered in play.** `J` toggles `showTriggers`: a translucent quad + direction arrow(s), green (armed) / red (disarmed), brightening to white on a fire flash (`updateTriggerDebug`, ticked in `animate`).
- **Editor:** right-click â†’ **Add trigger** (path-trigger width defaults to the road width at `t`). Drawn top-down as a **line** across the track with a direction arrow; select/drag to move (`t` or `x/z`), props edit width/height/rotation/direction; `ensureTrackIds`/`removeStaleSeams` prune dangling ones. **`J` is taken by the game's trigger debug view** â€” don't reuse it.

## Editor conventions (`js/editor.js`)

- Editor state lives in a single mutable `track` object (`TrackCore.cloneTrack(TrackCore.STARTER_TRACK)` initially).
- Undo/redo uses whole-track deep-clone snapshots (`undoStack`/`redoStack`, capped at `MAX_HISTORY`), not diffs. Every discrete mutation calls `pushUndo()` once *before* mutating, capturing pre-edit state. Continuous gestures (dragging a point, typing in a field) call `pushUndo()` once at gesture start, not per-tick, so one drag = one undo step â€” see `dragMutated` for how point-selection-without-dragging avoids recording a no-op step.
- Mesh regions: the JSON in `track` is authoritative (it is what undo snapshots and export read); `meshCache` holds a live geometry-js `Mesh` per asset purely to avoid reparsing. Rail edits mutate the live mesh and then `writeBackAsset()` immediately re-serializes it, so the two never drift. Undo/redo drops the cache entirely, since restored assets may differ.
- Dragging a mesh region in Edit mode moves it; shift+drag rotates it about its own placement origin instead (`dragging === 'meshRotate'`). The rotate branch records the offset between the drag start's angle-from-origin and the placement's current `rotation` at mousedown, then keeps applying that offset as the mouse moves, so the shape doesn't jump to face the cursor the instant the drag begins. Angle convention matches `TrackMesh.localToWorld`: `atan2(dz, dx)` from the placement's `(x, z)`, in degrees.
- The add-point/add-mesh popup (`#addPointMenu`) suppresses its own `contextmenu` event, separately from `topCanvas`'s. Right-click's `contextmenu` fires *after* mousedown, by which point the popup mousedown already opened is the frontmost element under the cursor -- so the browser hit-tests the popup, not the canvas beneath it, and `topCanvas`'s own handler never runs. Skipping this lets the native OS/browser context menu render on top of the custom one.
- The mode dropdown is `edit | create | rails`, with `E`/`C`/`R` shortcuts. All mode changes go through `setEditMode()` â€” it clears the abandoned create draft, drops the rail pick when leaving Rails mode, and syncs the dropdown, so the keys and the dropdown can't drift. Rails mode is modal on purpose: only mesh edges are pickable, so flagging a rail can't be confused with selecting anything else. Mesh regions are hit-tested *last* in edit mode, after every path handle, so a large region never steals a click from a control point drawn on top of it.
- Both modules expose a read-only `window.__editor` / `window.__game` handle for console debugging and the browser smoke tests, since ES modules leak nothing to the page.

## Game conventions (`js/track-game.js`)

- `buildTrack()` (re)generates the entire three.js scene/mesh from the current path list â€” importing new track JSON at runtime just calls it again rather than patching state incrementally.
- Track geometry/physics constants of note: `COLLISION_WALL_MARGIN` (defined in `track-core.js`). ### Surface-relative motion (why the ship survives a 90Â°-rolled wall)

The ship's facing and travel directions are **3D unit vectors in the surface tangent plane** (`physics.forward`, `physics.moveDir`), not world-yaw scalars, and its position (`physics.groundPos`) is fully 3D with Y authoritative. This is what lets it drive across, along, and through a vertical (or inverted) rolled surface. The earlier model stored a world-yaw `heading`/`velocityAngle` and integrated a purely horizontal velocity, then recovered the lateral offset `s` from X/Z by dividing by `cosÂ²(roll)` â€” a map that goes singular exactly as the road verticalizes near 90Â° roll, so the ship darted sideways and jammed into the walls. Two coupled root causes had to go, and both are gone:

- **Position recovery is exact at any roll, and segment selection is fully 3D.** `projectToSurface(sample, px, py, pz)` and `sampleTrack(x, y, z)`'s membership test take the *full 3D* point and dot it against the unit `edgeRight` â€” no division, no degeneracy. `sampleTrack` also *selects* the nearest centerline segment in 3D (project onto each segment in 3D, nearest 3D distance wins), which is what makes a **vertical loop** drivable: where the centerline itself goes straight up, the ascending and descending sides share an X/Z column, so an X/Z-footprint search could not tell top-of-loop from bottom-of-loop. Everything downstream that reasons about "along the segment/tangent" â€” the `SEGMENT_ALONG_TOL` foot test, `wouldOffEnd`/`offEnd`, and `corridorContains`'s `CORRIDOR_ALONG_TOL` â€” is 3D for the same reason. All of it **requires the ship's real Y**; passing the centerline's Y instead reintroduces the old singularity through the back door. On flat/banked (non-vertical-centerline) tracks the ship's Y â‰ˆ the surface Y, so 3D selection reduces to the old X/Z one â€” no behaviour change there; it only newly distinguishes things stacked in Y (loops, flyovers).
- **Velocity lives in the tangent plane** (`vel = moveDir * speed`, a full 3D vector), so on a bank it gains the vertical component that lets the ship climb toward a rolled edge, and on a loop it follows the centerline straight up and over. Steering rotates `forward` about the **surface normal** (`applyAxisAngle`), which is intrinsically correct through vertical and inverted sections â€” this is why the old `upSign` world-yaw flip hack is gone.
- **Grounded motion is magnetic** â€” while grounded, position is reprojected onto the surface every frame and no gravity is applied, so the ship adheres to the track like a slot car and holds a loop (even parked, even inverted at the top). It only goes ballistic by driving off an open curve's end (`offEnd`) or a bare mesh ledge; a closed loop path has neither, so it never falls off mid-loop.
- **`tangentize()` re-flattens `forward`/`moveDir` onto the surface, but only ONCE per frame** â€” in the steering/grip step, against that frame's own normal. Doing it again in the per-frame branches (or feeding the render basis back into `physics.forward`) re-projects the facing onto a second, slightly different plane; since every surface normal along the path is âŸ‚ the track tangent, those planes all share that tangent, and repeated projection slowly precesses the nose onto it (seen as the ship rotating to face along-track when parked near 90Â° roll). Keep tangentization to one projection per frame; the render block builds its display basis from a *copy* of `forward` and never writes it back.

`heading` survives only as a **derived** world azimuth of `forward` (`atan2(forward.x, forward.z)`), recomputed each frame purely for the top-down minimap. Airborne motion stays world-3D ballistic; `beginAirborne(vel3D)`/`landOnSurface(normal)` convert between the tangent-frame vectors and the ballistic (horizontal velocity + `verticalVel`) representation at the boundaries. **The whole thing is anchored on one invariant: on flat, unbanked ground every new expression reduces algebraically to the pre-rewrite one**, so normal driving is unchanged â€” that's the only correctness net, since this file has no automated coverage (browser-only) and the change must be playtested.
